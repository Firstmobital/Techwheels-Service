Claims Created — Not Yet Forwarded to TM

Business Logic Applied
⚙️ A claim with claim_status = "Created" means the dealer has initiated it on the TMPC portal but has NOT clicked Submit / Send to TM. TM cannot see, review, or approve it until submitted.
⚠️ Every day without submission risks policy window expiry — particularly for FSB claims which must be submitted within 15 days of job card closure.
💰 No reimbursement can occur until the claim progresses past "Created". These represent 100% preventable revenue loss if not acted on today.
SELECT COUNT(*) FROM ( SELECT id, 'wc' AS src FROM warranty_wc_data WHERE source_row_data->>'claim_status' = 'Created' UNION ALL SELECT id, 'updation' FROM warranty_updation_claim_data WHERE source_row_data->>'claim_status' = 'Created' UNION ALL SELECT id, 'part_wc' FROM warranty_part_wc_data WHERE source_row_data->>'claim_status' = 'Created' UNION ALL SELECT id, 'goodwill' FROM warranty_goodwill_data WHERE source_row_data->>'claim_status' = 'Created' UNION ALL SELECT id, 'fsb' FROM warranty_fsb_data WHERE source_row_data->>'claim_status' = 'Created' UNION ALL SELECT id, 'amc' FROM warranty_amc_data WHERE source_row_data->>'claim_status' = 'Created' ) t; -- Result: 87
🎯 Required Action: Service Manager to log into TMPC portal, locate all "Created" claims, and click "Submit to TM" today. Zero paperwork required — purely a portal action.

Business Logic Applied
📊 All three statuses represent terminal failure states — TM has closed these claims negatively. Unlike "Awaiting SOP", these cannot self-recover without dealer intervention.
📅 FSB rejections are entirely preventable — the 15-day JC-closure-to-submission policy limit is strict. The fix is operational: submit FSB claims the same day as JC closure, not days later.
💡 For WC/Updation rejections: check if vehicle was actually under warranty at service date. If incorrectly rejected, the appeal window must be used before it lapses.

Business Logic Applied
📄 Awaiting SOP Approval (53): TM has reviewed the claim and returned it requesting Standard Operating Procedure documentation — repair procedure photos, diagnostic scans, part removal evidence. Until the dealer uploads these, the claim is frozen. Inaction converts these to "Rejected."
✏️ Under Change (18): Either the dealer noticed a data error (wrong part number, incorrect labour code, mismatched chassis) or TM flagged an inconsistency and returned it for correction. The claim must be corrected and re-submitted.
⏱️ These are time-sensitive but salvageable. Claims in "Awaiting SOP" or "Under Change" for more than 5 business days without dealer response typically auto-reject at TM end.
SELECT source_row_data->>'claim_status' AS status, COUNT(*) AS cnt FROM ( SELECT source_row_data FROM warranty_wc_data UNION ALL SELECT source_row_data FROM warranty_updation_claim_data UNION ALL SELECT source_row_data FROM warranty_part_wc_data UNION ALL SELECT source_row_data FROM warranty_goodwill_data ) t WHERE source_row_data->>'claim_status' IN ('Awaiting SOP Approval', 'Under Change') GROUP BY 1; -- Awaiting SOP Approval: 53 | Under Change: 18 → Total: 71
📌 Note: FSB table excluded from this alert — FSB claims don't use "Awaiting SOP" status. The 53+18 records are spread across WC, Updation, Part WC, and Goodwill claim types.

Business Logic Applied
📑 The warranty_claim_settlement_report_data table is the SAP settlement register — each row is a line item (individual part or labour charge) within a settled warranty claim. The posting_document_number field contains the SAP FI document reference generated when TM posts the credit to the dealer's account.
🔗 Key fields in this table: prowac_no (claim reference), job_card_number_number (linked JC), dealer_invc_no (dealer invoice), posting_date_yyyy_mm_dd, posting_document_number. When posting_document_number is blank, the accounting entry has NOT been made.
💳 1,137 unposted lines means approx. 27.7% of settlement register is in an accounts-pending state. Multiple lines can belong to one PROWAC claim (parts + labour = 2+ lines per claim), so unique pending claims is fewer than 1,137.
⚡ This is an accounts department action, not service team. SAP posting is typically done by TM's regional accounts team — dealer needs to follow up with TM accounts for early posting to receive credit in the current accounting cycle.
SELECT COUNT(*) FILTER (WHERE source_row_data->>'posting_document_number' = '') AS sap_pending, COUNT(*) FILTER (WHERE source_row_data->>'posting_document_number' != '') AS sap_posted, COUNT(*) AS total_lines FROM warranty_claim_settlement_report_data; -- sap_pending: 1,137 | sap_posted: 2,973 | total: 4,110
📊 Table also contains: prowac_no · tml_reference_number · sap_claim · ndp (part price) · labour_chgs · spl_labour_chgs · no_off_complains. Cross-join with warranty_wc_data on prowac_no to identify which specific claims these lines belong to.

AMC Settlement Workflow (actual)
1️⃣Dealer creates AMC claim → status: Created/Sent to TM
2️⃣L1 officer approves, settled_tm_amount pre-computed → status: Approved By L1
3️⃣L2 officer confirms → status: Approved by L2
4️⃣🚫 BLOCKED HERE: dealer must raise invoice in SAP → dealer_invoice_no gets filled
5️⃣TM processes payment → settlement_dt / payment_dt filled → money credited
Business Logic Applied
🔍 AMC reimbursement requires TWO independent actions: (A) TM approval and (B) dealer invoice. TM has completed action A for all 91 claims — the settled_tm_amount is already computed in the database. The dealer simply hasn't taken action B (raised the invoice) despite TM being ready to pay.
💰 From sample data, individual AMC claim amounts range Rs. 3,186 – Rs. 11,963 per claim. At an average of ~Rs. 5,800 per claim: 91 claims × Rs. 5,800 = approx. Rs. 5.3 Lakhs uncollected.
📋 The dealer_invoice_no field in warranty_amc_data tracks the dealer's invoice number. For all 91 flagged records, this field is an empty string while claim_status shows TM approval. This cross-field inconsistency is the precise signal used for this alert.
⚡ This is uniquely AMC-specific — not applicable to WC/FSB/Goodwill claims which use a different settlement pathway (direct posting via settlement report).
SELECT source_row_data->>'claim_status' AS approval_level, COUNT(*) AS pending_invoice_count, source_row_data->>'amc_type' AS amc_type FROM warranty_amc_data WHERE source_row_data->>'claim_status' IN ('Approved By L1', 'Approved by L2') AND source_row_data->>'dealer_invoice_no' = '' GROUP BY 1, 3 ORDER BY 2 DESC; -- Approved By L1 + empty invoice: 8 -- Approved by L2 + empty invoice: 83 -- Total: 91 AMC claims ready for dealer invoicing
🚀 Required Action: Accounts/Warranty team to log into SAP, raise dealer invoices for all 91 AMC PROWAC references, and update dealer_invoice_no in the TMPC portal. Estimated cash recovery: ~Rs. 5.3 Lakhs.
