## Database Authority Hierarchy

There are three tiers, in this order:

1. `supabase/backups/full_metadata.sql` — **primary**. Schema/object metadata truth.
2. `local_folder/backups/full_database.sql` — **secondary**. Full database truth (schema + row data).
3. `local_folder/backups/chunks/full_database.sql.part_*` — **mirror**, not a separate tier. Exact split of file #2, used only when direct access to #2 is blocked by size/indexing/token/context limits.

Resolution order for any database question:

1. Is the question about schema shape (tables, columns, types/enums, constraints, indexes, views, materialized views, functions/RPCs, triggers, RLS, policies, grants/privileges, extensions, comments)? Use `full_metadata.sql` first.
2. Is the question about row data, seed/lookup/master data, default business configuration, or do you need complete database evidence (schema + data together)? Use `full_database.sql`.
3. Can't read `full_database.sql` directly (size/indexing/token/context limit)? Read the chunk parts in `local_folder/backups/chunks/` in order — they are an exact continuous mirror of the same file, never a different version.
4. If `full_metadata.sql` and `full_database.sql` ever disagree on schema shape, `full_metadata.sql` wins (it is the dedicated schema authority and is regenerated independently).

## Purpose of Each Truth File

### 1. `supabase/backups/full_metadata.sql` (primary)

Schema-only `pg_dump` (`--schema-only --no-owner`, grants/privileges included — `--no-privileges` is never used). Generated by `scripts/backup-metadata.sh`.

Use it first for: tables, columns, types/enums, constraints, indexes, views, materialized views, functions/RPCs, triggers, row-level security, policies, grants/privileges, extensions, and comments.

Why it is primary: it is the smallest, fastest-to-read, most current representation of pure schema shape. It contains no row data, so it is cheap to search end-to-end and is regenerated independently of the (much larger) full data dump, making it the most reliable single source for "does this column/policy/function exist, and what is its exact definition?"

### 2. `local_folder/backups/full_database.sql` (secondary)

Full `pg_dump` (schema + data, `--create --clean --if-exists`). Generated by `scripts/backup-full-db.sh`.

Use it when you need: row data, seed data, lookup/master data, default business configuration, or complete database evidence (i.e. you need to see actual values, not just shape). It also contains the full schema, so it remains a valid cross-check against `full_metadata.sql`, but it is not the first place to look for schema-only questions because of its size.

### 3. `local_folder/backups/chunks/full_database.sql.part_*` (access mirror, fallback only)

Exact byte-for-byte split of `full_database.sql` into ~19MB parts (`full_database.sql.part_000`, `part_001`, ...). Generated by the same `scripts/backup-full-db.sh` run that produces `full_database.sql` — never generated independently and never edited by hand.

Why chunks exist: `full_database.sql` can exceed file-size, indexing, token, or context limits for some tools/agents. The chunk parts let those tools read the same content in smaller pieces, in order, as a drop-in continuous mirror. Treat the parts as pages of one file, not as separate dumps, snapshots, or versions — a part being older/newer than another part should never happen since they are always regenerated together as one set.

## Database Inspection Rules for AI Agents

1. Always check the authority hierarchy above before answering any question about the database. Schema questions go to `full_metadata.sql` first; data questions go to `full_database.sql`.
2. If `full_database.sql` cannot be read directly due to size/token/context limits, read the `local_folder/backups/chunks/full_database.sql.part_*` files in numeric order as a continuous mirror — never skip parts, never treat a part as a standalone fact.
3. **Never invent database objects.** Do not assume a table, column, type, enum, constraint, index, view, materialized view, function/RPC, trigger, policy, grant, or extension exists (or has a particular shape) unless it is actually present in `full_metadata.sql` or `full_database.sql`. If it is not in either file, say so explicitly instead of guessing.
4. Do not infer row-level security or policy behavior from application code alone — verify against `CREATE POLICY` / `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` statements in `full_metadata.sql`.
5. If a dump and a doc disagree, the dump wins (this mirrors the existing rule in `docs/shared/reference/DB_TRUTH_PROTOCOL.md`).
6. For DB changes in progress (proposed/applied/verified but not yet captured in a fresh dump), use the composite truth described in `docs/shared/reference/DB_TRUTH_PROTOCOL.md`: latest dump + promoted migrations listed in `supabase/evidence/post_dump_verified_promotions.md`.
7. Regenerate dumps with the scripts below rather than hand-editing dump files. Hand edits will be silently overwritten on the next regeneration and are never authoritative.

## Regeneration Commands

Both scripts read credentials only from environment variables (`SUPABASE_DB_HOST`, `SUPABASE_DB_PORT`, `SUPABASE_DB_NAME`, `SUPABASE_DB_USER`, `SUPABASE_DB_PASSWORD` — see `.env.example`). Nothing is hardcoded.

Generate/refresh the primary metadata truth file:

```bash
export SUPABASE_DB_HOST=... SUPABASE_DB_PORT=... SUPABASE_DB_NAME=... \
       SUPABASE_DB_USER=... SUPABASE_DB_PASSWORD=...
scripts/backup-metadata.sh
```

Generate/refresh the secondary full-database truth file and its chunk mirror:

```bash
export SUPABASE_DB_HOST=... SUPABASE_DB_PORT=... SUPABASE_DB_NAME=... \
       SUPABASE_DB_USER=... SUPABASE_DB_PASSWORD=...
scripts/backup-full-db.sh
```

`scripts/refresh_authoritative_dump.sh` still works and simply calls `scripts/backup-full-db.sh` — kept for backward compatibility with existing docs/automation.
