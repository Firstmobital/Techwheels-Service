# Repository AI Operating Contract

**Last Updated:** 2026-06-29
**Status:** Active — this file is the single, generic entry point for every AI agent/tool working in this repository.

## 1) One Contract Rule

1. This repository uses exactly **one** generic, vendor-neutral AI-agent operating contract: this file (`.instructions.md` at the repository root).
2. Agents must **not** create vendor-specific instruction files, including (not limited to): `CLAUDE.md`, `CURSOR.md`, `.cursor/rules`, `copilot-instructions.md`, `GEMINI.md`, `CODEX.md`, or any other tool-specific AI doc.
3. Do not create `docs/ai-context.md` or any second generic contract file. If a new tool/workflow needs guidance, update this file instead of creating a parallel one.
4. If you find a vendor-specific or duplicate AI instruction file anywhere in the active repo (outside `local_folder/Reference/`, which holds archived copies of other repos and is out of scope), do not delete it immediately — check it for unique instructions, merge anything useful here, then report it as deprecated/removable rather than silently deleting it.

## 2) Task Decision Framework

Classify every request before reading anything, then consult only the authorities listed for that classification. Avoid unnecessary reading — do not read the entire repository by default.

1. **Code** (application/business logic, UI): `src/` or `mobile/src/` directly -> `docs/shared/README.md` (architecture/business-domain reference) -> `docs/shared/reference/CURRENT_STATE.md`.
2. **Database** (schema, RPCs, policies, data): `docs/shared/reference/DATABASE_TRUTH.md` -> the dump it points to (`supabase/backups/full_metadata.sql` for schema, `local_folder/backups/full_database.sql` for data) -> relevant code -> `docs/shared/reference/DB_CHANGE_PROTOCOL.md`/`DB_CHANGE_LEDGER.md` if proposing a change.
3. **Documentation** (creating/moving/editing any doc): `docs/STRUCTURE_GUIDE.md` -> search for an existing doc on the same subject (Section 5, New File Decision Tree).
4. **Implementation Plan**: `docs/Implementation_plans/INDEX.md` and `IMPLEMENTATION_TRACKER.md` for the relevant platform -> `docs/STRUCTURE_GUIDE.md` Sections 3.5/5/7 for lifecycle and naming.
5. **Architecture** (module boundaries, route/data-flow design): `docs/shared/README.md` (architecture sections) -> `docs/shared/reference/MODULE_ROUTE_CONTRACT.md` / `ROUTE_STRATEGY_DECISION.md` -> code.
6. **Configuration** (env vars, build/runtime config): `.env.example` (placeholder convention only — never read real secrets from `.env.local`) -> `package.json` scripts -> the config file in question.
7. **CI/CD** (workflows, PR gating, deploy automation): `.github/workflows/*.yml` -> `.github/CODEOWNERS` -> `.github/pull_request_template.md`.
8. **Infrastructure** (Supabase project/edge-function deployment, hosting): `.github/workflows/deploy-edge-functions.yml` -> `supabase/functions/` -> `docs/shared/reference/DATABASE_TRUTH.md` for any DB-adjacent infra question.
9. **Mixed** (spans more than one category above): consult every authority listed for each category involved, plus `docs/shared/reference/SYNC_PROTOCOL.md` (cross-cutting docs-sync rule for any change spanning code/schema/docs) and `docs/DOCS_IMPACT_MATRIX.md` (maps code-change areas to required doc updates).

If a request doesn't fit cleanly, classify it as Mixed rather than guessing. Under-classifying causes missed authorities; reading the whole repository "to be safe" is also wrong — read only what the matching classification points to.

## 3) Repository Authority Resolution

When sources could conflict, resolve authority in this order:

1. **Visible user instructions** — what is being asked right now, in this conversation, takes precedence over any standing doc.
2. **Current repository state** — the actual code/config/files as they exist now.
3. **Authoritative governance** — `docs/STRUCTURE_GUIDE.md` (documentation placement) and this file (AI-agent behavior): the standing rules for how to work in this repo.
4. **Task-specific truth** — the dedicated authority file for the task's domain (see the Task Decision Framework, Section 2, for the full mapping). For example: Documentation -> `docs/STRUCTURE_GUIDE.md`; Database -> `docs/shared/reference/DATABASE_TRUTH.md`; Implementation status -> the relevant plan under `docs/Implementation_plans/`; Runtime behavior -> the code itself; Business logic/current architecture -> `docs/shared/README.md` and `docs/shared/reference/CURRENT_STATE.md`.
5. **Historical information** — change logs, ledgers, completed/archived plans, and superseded dumps (e.g. `docs/shared/active/CHANGE_LOG.md`, `docs/shared/reference/DB_CHANGE_LEDGER.md`, `docs/Implementation_plans/completed/`, the historical dumps listed in `DATABASE_TRUTH.md`). Useful for context; never authoritative over levels 1–4.

"Task-specific truth" means: of all the standing governance docs in this repo, the one whose stated scope is the exact domain of the current task. When two sources at the same level disagree, the more domain-specific one wins (e.g. on a schema question, `DATABASE_TRUTH.md` outranks a general architecture doc).

## 4) No Assumption Protocol

Before proposing or making any change:

1. **Search first** — look for existing implementation, docs, or scripts covering the request before writing anything new.
2. **Inspect the existing implementation** — read the actual code/schema/doc involved, not a summary or memory of it.
3. **Determine the authoritative source** — using Sections 2 and 3 above.
4. **Only then propose changes.**

Never invent, when an authoritative source already exists and doesn't show it:
- database tables, RPCs/functions, columns, triggers, policies, or permissions (Section 8 item 5 already states this rule for the database specifically — this generalizes it)
- routes, modules, or screens
- documentation files or sections
- scripts or automation

If something can't be confirmed in an authoritative source, say so explicitly instead of guessing.

## 5) New File Decision Tree

Before creating any new file (code, doc, script, or config):

1. Search the repository for an existing file that already owns this content or purpose.
2. If an existing file owns it -> update that file. Do not create a parallel one.
3. Else, if the content is documentation -> determine placement using `docs/STRUCTURE_GUIDE.md` (state/scope/intent), then create it there.
4. Else (not documentation) -> place it following the existing convention for that file type in this repo (e.g. `scripts/` for automation, `supabase/migrations/` for schema changes), then create it.
5. Report the search performed and why no existing file qualified, before creating anything.

Section 7 item 3 defers to this rule for documentation specifically; item 4 adds a documentation-only reporting step on top of it.

## 6) Repository Minimalism Rule

Prefer, in order: updating > extending > refactoring > creating. Avoid duplicating or forking a parallel version of documentation, scripts, configuration, or instruction/contract files — one authoritative source per concern is the goal. This generalizes Section 1 item 3 (no second AI contract) and Section 7's documentation-duplication rule to every file type in the repo.

## 7) Documentation Placement Governance

1. Before creating, moving, renaming, or editing any documentation, read `docs/STRUCTURE_GUIDE.md` first — it is the placement authority for this repo.
2. Classify every doc before acting on it:
   - **state:** truth or implementation
   - **scope:** shared, web, or mobile
   - **intent:** reference, evidence, runbook, active, or governance
3. Before creating a new doc, follow the New File Decision Tree (Section 5) — search for an existing owner first; update it instead of duplicating.
4. Report the placement rationale (state/scope/intent and target path) before creating new documentation.
5. Do not leave stale links: when a doc is moved, update every inbound reference in the same change and remove the old path only after links are updated.

## 8) Database Truth Authority

1. Schema/object metadata (tables, columns, types/enums, constraints, indexes, views, materialized views, functions/RPCs, triggers, RLS, policies, grants/privileges, extensions, comments): `supabase/backups/full_metadata.sql` — **primary**.
2. Row data, seed/lookup/master data, default business configuration, full database evidence: `local_folder/backups/full_database.sql` — **secondary**.
3. `local_folder/backups/chunks/` — exact access mirror of #2 only, never a separate version.
4. Full hierarchy, purpose of each file, and AI inspection rules: `docs/shared/reference/DATABASE_TRUTH.md`.
5. **Never invent database objects.** If a table, column, function, policy, trigger, or permission is not present in the authoritative dump files, say so explicitly instead of guessing.

## 9) Code Truth

1. The local repository is code truth. Do not assume application behavior that isn't visible in the code or in the authoritative dump files above.
2. Do not modify application logic, business logic, or runtime behavior as a side effect of documentation, database-truth, or instruction-file work unless explicitly asked.

## 10) Implementation Plan Automation

**When you ask for an implementation plan, I will automatically:**

1. ✅ Create a new plan file in `docs/Implementation_plans/[PLAN_NAME].md`
2. ✅ Fill in all sections with complete, actionable content (no placeholders left)
3. ✅ Set up activity tracker with real-time status indicators
4. ✅ Update `docs/Implementation_plans/INDEX.md` with the new plan
5. ✅ Organize tasks into logical phases
6. ✅ Include risk assessment, success criteria, and detailed breakdowns

**No manual template copying or editing needed.**

### How to Request a Plan

Use natural language. No special format required. Examples:

- "Create an implementation plan for mobile app push notifications"
- "I need a plan to migrate the database to PostgreSQL"
- "Set up a plan for user analytics tracking"
- "Create a plan to implement dark mode support"

### Activity Tracker Updates

The activity tracker in each plan uses these symbols:
- ⏳ PENDING — Task not started
- 🔄 IN PROGRESS — Currently working
- ✅ COMPLETED — Finished
- ❌ BLOCKED — Waiting on external dependency

Update these in the plan file as work progresses. I will ask for updates or check status upon request.

### Plan File Structure

All plans include:
- Executive Summary with risk level & duration
- Objectives & context
- Implementation tasks organized into phases
- Real-time activity tracker
- Detailed task breakdowns with acceptance criteria
- Risk assessment matrix
- Success criteria & sign-off section
- Notes & lessons learned section

### Dashboard & Navigation

- **Quick Overview:** Check `docs/Implementation_plans/INDEX.md` for all active/completed plans
- **Full Details:** Click on individual plan file for complete information
- **Progress Updates:** Edit activity tracker in each plan file as work advances

### No Reminders Needed

You don't need to remind me of this. Any request for a plan will trigger automatic creation with full structure and content. Just ask naturally.
